00 Purpose

This guide equips Solution Engineers to build working rapid prototypes using GitHub Copilot. The goal isn't to become software engineers. It's to close the gap between knowing what a solution should do and showing it working. A credible 1–3 day prototype changes the customer conversation.

Prototypes built in the field don't need to be production-ready. They need to be real enough to feel true: working API calls, real data shapes, an interface a customer can click through. GitHub Copilot makes that bar achievable without a deep engineering background.

01 Mindset Shift

Prototyping with AI assistance is a different skill than traditional coding. You already have the hardest part: customer context and domain knowledge. Here's how to pair that with GitHub Copilot effectively:

🏗️

You're the architect

GitHub Copilot writes code. You decide what to build and whether the output is right.

🔁

Iteration > perfection

A working prototype in 2 hours beats a polished plan that never runs.

🗣️

Context is leverage

The more context you give GitHub Copilot, the better the output. Your domain knowledge is the asset.

Common trap Don't spend time trying to get the prototype fully aligned with what the customer wants in production. Position it when meeting with the customer as: "We know when we rapid prototype that it's going to be wrong. But we are demonstrating the capability to ground the discussion, and we will iterate with you as we co-develop."

02 Setup

Prerequisites

  • VS Code installed (latest stable)
  • GitHub Copilot + Copilot Chat extensions installed and licensed — confirm with your manager
  • Python 3.11+ installed
  • Azure CLI installed and authenticated (az login)
  • Access to an Azure subscription with Microsoft Foundry provisioned

Key Extensions

GitHub Copilot GitHub Copilot Chat Python Azure Tools REST Client
Use Plan Mode In GitHub Copilot chat, toggle to Plan mode. This is ideal for planning your project before coding begins. Use it for your starting prompt.
Enable Agent Mode In VS Code settings, set github.copilot.chat.agent.enabled: true. This is the most powerful mode. It scaffolds entire projects, runs terminal commands, and edits multiple files at once.

03 Prototyping Workflow

Use this loop for every prototype. The full cycle should take 2–4 hours for most SE demos.

01

Define the demo moment first

Write one sentence: "The customer will see ___." Work backwards from that. One impressive thing end-to-end beats three half-finished things.

02

Drop in your reference files

Add your OCTO starter template (sample agent file), .env, and requirements.txt to the project folder before opening Copilot. These are your context — Copilot builds from them, not from scratch.

03

Send your starting prompt, and plan first

Use the template in Section 04. Ask GitHub Copilot to output a plan before writing any code, then wait for your approval. The plan is your cheapest chance to course-correct.

04

Get it running

Follow GitHub Copilot's setup instructions. If something fails, paste the full terminal error back into GitHub Copilot Chat. Repeat until python app.py gives you something visible.

05

Use synthetic data as needed

Use synthetic customer-like inputs to reduce cycle time. Your SE context — knowing what realistic data looks like for this customer — is what makes this step fast.

06

Polish the demo path only

Identify the exact 5-minute click path the customer will see. Harden that path. Don't polish anything else.

04 Your Starting Prompt

Fill in the two highlighted fields. The rest is already tuned.

Copilot Chat — Agent Mode
I'm building a rapid prototype for [customer / industry] that demonstrates [one-sentence use case].

Reference files are in the repo: sample_agent.py, .env, requirements.txt.
Use Python FastAPI with an HTML/CSS/JS UI. Target deployment: Azure Web App.
Follow the patterns in the sample implementation.

Before writing any code, show me a proposed plan and list any assumptions.
Wait for my approval before proceeding.
That's it Two blanks. The prompt asks for a plan first so you can steer before any code is written — that conversation is where you shape the output. If you want a specific UI (e.g., a dashboard with run history), add one sentence at the end describing it.
OCTO Starter Templates The OCTO maintains starter templates — a sample agent file, .env shape, and requirements.txt — for each common demo pattern. Clone one before starting. It gives GitHub Copilot a concrete reference to follow rather than inventing its own approach.

05 Iterating with Copilot

Once the prototype is running, use these short prompts to move fast without breaking things.

You want to…Say this
Fix a crash Paste the full terminal error. Nothing else needed.
Change a behavior "Instead of X, do Y. Don't change anything else."
Add something "Add [feature] to [file name]. Keep all existing logic intact."
Demo-proof it "If the API call fails, fall back to the sample data in the attached file."
Understand something "Explain what this function does in plain English."
Polish the UI "Make the UI look clean and professional. Dark theme, Microsoft blue accents."
Always scope your changes Vague prompts cause unintended rewrites. Name the file or function you want changed. "Fix the submit button in main.py" is better than "fix the form."

08 Azure Auth & Deployment

This is consistently the hardest part of field prototypes. The right setup depends on which Azure services you're connecting to — but the pattern is almost always the same: Managed Identity + role assignment + DefaultAzureCredential in code.

Before you start Auth setup requires Azure permissions your subscription may not grant by default. If you hit a wall, loop in your OCTO contact or a cloud architect — don't guess your way through RBAC. Getting this wrong can silently fail or expose resources.

The Standard Pattern (3 Steps)

Step 1
Enable Managed Identity on Web App
→
Step 2
Assign Role on Target Service
→
Step 3
Use DefaultAzureCredential in Code
Step 1 — Enable system-assigned managed identity
az webapp identity assign \
  --name <your-app-name> \
  --resource-group <your-rg>
Step 2 — Assign the role (example: Azure OpenAI)
az role assignment create \
  --assignee <principal-id-from-step-1-output> \
  --role "Cognitive Services OpenAI User" \
  --scope /subscriptions/<sub>/resourceGroups/<rg>/providers/\
Microsoft.CognitiveServices/accounts/<openai-resource-name>
Step 3 — Use DefaultAzureCredential in Python
from azure.identity import DefaultAzureCredential

credential = DefaultAzureCredential()

# Works for Azure OpenAI, AI Foundry, Key Vault, Storage, etc.
# Same code works locally (uses az login) and on Azure (uses Managed Identity)
Why this works everywhere DefaultAzureCredential automatically uses your local az login session during development and switches to Managed Identity when deployed — so the same code runs in both environments without any changes.

Role Reference by Service

The role name is what varies by service. This covers the most common prototype scenarios.

Local vs. Deployed — Quick Reference

EnvironmentHow auth worksWhat to do
Local dev Azure CLI credential Run az login. DefaultAzureCredential picks it up automatically.
Azure Web App Managed Identity Enable system-assigned MI on the Web App. Assign roles per the table above.
CI/CD pipeline Service Principal Not needed for most prototypes — skip until you need it.

Locking Down the Web App (User Access)

If the prototype should only be accessible to specific people, use Easy Auth — don't add custom auth code.

# Azure Portal: Your Web App → Authentication → Add identity provider → Microsoft
# Set "Restrict access" to "Require authentication"
# Use your Microsoft tenant, or the customer's if they'll be accessing it directly
Let GitHub Copilot generate the auth setup for you Once your app is running locally, ask: "Set up DefaultAzureCredential for [service]. I'm deploying to Azure Web App with system-assigned Managed Identity. Show me the Python code and the role assignment CLI command." It will generate both for the specific service.

09 What to Avoid

Anti-patternWhy it hurtsInstead
Over-scoping You run out of time. Nothing works on demo day. One strong scenario, end-to-end.
Never running the code You discover the problem in front of the customer. Run after every major change.
Trusting GitHub Copilot blindly It confidently generates plausible-but-wrong code. Test every piece of logic against real inputs.
No fallback for the demo API timeouts kill live demos. Add a mock data fallback. Always.
Hardcoding secrets Keys end up in screenshots, repos, or slide decks. Use .env locally. Use Key Vault or App Settings in Azure.
Guessing through auth errors Hours lost, possible misconfiguration. Use Section 08.
